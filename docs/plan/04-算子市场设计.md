# 04 · 算子市场(Operator Marketplace)设计

> 关联:[03-架构设计](03-架构设计.md) · [00-实现任务清单](00-实现任务清单.md) · [02 集成调研](02-data-juicer-集成与并发调研.md)
> 数据来源:`data-juicer/docs/Operators.md` + `docs/operators/**`,经 `backend/scripts/build_operator_catalog.py` 解析为 `backend/app/data/operators_catalog.json`。

## 1. 目标与范围

把 data-juicer 的**全部 212 个算子**做成一个可浏览、可检索、可按场景/模态/算力分面筛选的"算子市场",并在新建数据加工任务时直接选用。市场必须解决一个核心现实:**算子需要的"模型/算力"各不相同**——有的纯 CPU 即可,有的要 LLM API,有的要 GPU/HuggingFace 模型。市场要把这件事显式化,**把算子路由到合适的执行后端**(用户原话"使用合适的模型处理任务"),并对当前环境跑不了的算子优雅降级(展示但禁用,标注前置条件)。

范围:
- **做**:全量算子目录(解析产物)、目录查询/分面 API、算子市场前端页、与加工任务的衔接、资源前置校验。
- **本期不做**:GPU 节点接入、HuggingFace 模型下载管线、vLLM 服务编排(只预留接口与标注)。

## 3. 算子目录数据模型

单一事实源:构建脚本产出的 `operators_catalog.json`(随后端发布,启动时加载进内存)。每个算子节点:

```jsonc
{
  "name": "clean_html_mapper",           // DJ 算子注册名(= dj-process process 项的 key)
  "category": "mapper",                   // 8 类之一
  "summary_en": "...", "summary_zh": "从文本示例中清除HTML代码…",
  "desc_en": "...",  "desc_zh": "…",       // 详情页完整段落(可空)
  "modality": ["text"],                   // text/image/audio/video/multimodal,可空=通用
  "compute": "cpu",                        // cpu | gpu
  "frameworks": [],                        // api | vllm | hf 的子集
  "stability": "stable",                   // alpha | beta | stable
  "resource_class": "cpu",                 // 派生:api_llm>vllm>hf_model>gpu>cpu
  "params": [{"name":"min_len","type":"<class 'int'>","default":"10","desc":"…"}],
  "example": "CleanHtmlMapper()",          // 详情页首个 python 示例
  "reference": null,                        // 论文/库引用(可空)
  "detail_page": "operators/mapper/clean_html_mapper.md",
  // —— 语义增强 ——
  "scenario_group": "文本清洗",            // 17 个用户侧场景之一(LLM 标注,值受枚举约束)
  "zh_label": "HTML标签清洗",              // 4-12 字界面短名(LLM)
  "zh_usage_tip": "去除文本中残留的HTML标签,适合处理网页抓取的原始语料。",  // 何时用(LLM)
  "runnable": "ready",                     // ready|needs_api|needs_media|needs_compute(代码推导)
  "recommend": true                         // = LLM 推荐 且 runnable==ready
}
```

> **增强字段的来源(Rule 7 取舍)**:`runnable` 由代码从 `resource_class` + 模态确定性推导(纯逻辑);`scenario_group / zh_label / zh_usage_tip` 采用 LLM 标注结果(经校验:212/212 全覆盖、命名零误、场景值全部落在 17 枚举内,且对 `detect_character_*` / `calibrate_query` 等边界算子比纯规则更准);`recommend = LLM推荐 且 runnable==ready`(推荐视图不出现跑不了的算子)。LLM 标注以 `backend/scripts/operator_marketplace_meta.json` 形式纳入构建管线,缺失项回退到确定性规则,**可复现**。

### `runnable` 计算规则(模型/算力路由的核心)

| runnable | 条件 | 含义 / 市场表现 |
|----------|------|----------------|
| `ready` | `resource_class=cpu` 且模态 ⊆ {text, 通用} | 现有 dj-process 引擎**直接可跑**,正常"加入任务" |
| `needs_media` | `resource_class=cpu` 但模态含 image/video/audio/multimodal | 算子能跑但**受管数据集是文本 jsonl**,无媒体文件 → 标注"需媒体数据",禁选 |
| `needs_api` | `resource_class=api_llm` | 需在 `.env` 配 `OPENAI_*`;**已配则可选**,未配则提示去配置 |
| `needs_compute` | `resource_class ∈ {hf_model, gpu, vllm}` | 当前无 GPU 环境 → 标注"需要算力(暂不可用)",禁选 |

## 4. 后端 API 设计

复用现有 FastAPI 风格(`{data, success}` 信封、camelCase 出参)。新增 `app/services/operator_catalog.py` 改为**加载 JSON + 内存索引**,新增 `app/api/v1/operators.py`。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/operators` | **保持兼容**:返回旧 9 字段形态(`name/category/label/description/params`),但 `params` 经类型归一可被前端表单消费。默认只回 `runnable=ready` 的算子,供加工页下拉。 |
| GET | `/operators/catalog` | 市场主接口。查询参:`scenario` / `category` / `modality` / `resourceClass` / `runnable` / `recommend` / `keyword`(匹配 name+中文摘要)/ `current` / `pageSize`。返回分页 + 各分面的计数(facets),供左侧筛选栏。 |
| GET | `/operators/catalog/meta` | 返回 `meta`(总数、各维度分布、场景枚举)——驱动筛选项与概览卡。 |
| GET | `/operators/{name}` | 单算子详情(全字段:参数表 + 示例 + 引用 + 详情页链接)。 |

**参数表单归一**:DJ 参数 `type` 是 Python 类型串(`<class 'int'>`、`typing.Optional[str]`…)。后端提供 `_ui_field(param)` 把它映射为前端字段类型:`int/float→number`,`bool→switch`,`str→text`,有限枚举(如 chinese_convert 的 mode)→`select`;并跳过 `args/kwargs` 这类无意义项。归一只在 `/operators`(加工页直用)与详情里做,`catalog` 原样透出 DJ 参数表。

**加工任务校验升级**(`POST /jobs`):
1. 算子名 ∈ 全量目录(原来是 9 个白名单);
2. 逐个算子 `runnable` 必须为 `ready` 或(`needs_api` 且后端检测到 `OPENAI_*` 已配),否则 400 返回清晰原因:`算子 image_aesthetics_filter 需要 GPU/模型,当前环境不可执行`。这一步就是"把算子路由到合适后端"的守门。

## 5. 模型 / 算力路由

```
              ┌─────────────────────── 算子 resource_class ───────────────────────┐
   cpu(122)            api_llm(28)           hf_model(28)/gpu(30)/vllm(4)
     │                    │                            │
  本机 dj-process     dj-process + .env            (本期不接)
  子进程(已实现)      OPENAI_* 走 LLM API          GPU 节点 / vLLM 服务
     │                    │                            │
  runnable=ready      runnable=needs_api          runnable=needs_compute
  立即可用            配置后可用                   市场展示+禁用+"需要算力"
```

- **M0 可运行集 = 89 个 `ready` 算子**:覆盖文本清洗/质量过滤/去重/中文处理/隐私脱敏/数据选择,足以支撑"语料治理"主线。
- **api_llm 引导**:市场详情页对 needs_api 算子给"需要配置 LLM"提示 + 一键跳"设置"(写 `.env` 的 `OPENAI_*`);后端 `engine` 已支持 LLM 失败回退启发式。
- **needs_compute / needs_media**:展示但置灰,卡片角标说明前置条件;为将来 GPU 节点预留 `resource_class → executor` 映射点(目前只有 `cpu/api_llm → 本地 dj-process`)。

```
数据加工 ▾
  ├─ 加工任务   /processing/jobs   (现有 ./processing)

- **卡片**:中文短标签 + 英文名 + 类型/模态标签 + 一行中文摘要 + 状态角标(🟢可运行 / ⚙️需配置 / ⚠️需算力 / 🖼需媒体)+ ⭐推荐。`[查看]` 开详情抽屉,`[+]` 加入"待编排算子"。

## 7. 与现有加工流程 / 数据集仓库集成

- 加工页(`processing`)的算子来源从 `GET /operators`(仍兼容)切到"市场已选";未走市场也能在加工页内直接多选(下拉只列 `ready`)。
- 加工执行链路**不变**:Job 读入版本 → `engine.build_config` 生成 DJ YAML → 子进程 `dj-process` → 产出新版本(见 03)。市场只改"选什么算子",不改"怎么跑"。
- 血缘/版本不受影响:算子目录是**无状态参考数据**,不入库、不产生版本。

## 9. 风险与取舍

- **目录漂移**:catalog 由 data-juicer 文档解析而来,DJ 升级后需重跑 `build_operator_catalog.py`。取舍:接受"构建期快照",优于运行时反射(后端 py3.12 无法直接 import DJ 的 py3.11 算子)。脚本幂等,刷新成本低。
- **场景/标签的 LLM 标注 vs 规则**:本期采用 LLM 标注(经完整校验后纳入 `operator_marketplace_meta.json`),对边界算子比纯规则更准;同时保留确定性规则作为缺失项回退,逻辑字段(runnable)始终由代码推导。取舍:LLM 标注质量更高且已固化为可复现的数据文件,审计成本可控;若 DJ 新增算子而未及时标注,自动回退规则不致开天窗。
- **needs_media 的本质**:受管数据集当前是文本 jsonl,图像/音视频算子即便是 CPU 也无文件可处理。标注禁用而非隐藏,保留"能力可见性",为将来多模态数据接入留位。
- **`/operators` 兼容**:旧前端加工页依赖 9 字段形态,新接口保留该形态(只是数据源换成全量里的 ready 子集 + 参数归一),避免破坏现有加工流程。
```
