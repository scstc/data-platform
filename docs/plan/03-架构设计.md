## 0. 核心设计思想:一个引擎,三个名词

20 项需求里约 18 项可以归约到**三个实体 + 一个动词**,而不是 20 个互不相干的功能:

| 实体 | 角色 | 承载的需求 |
|------|------|-----------|
| **DataSource** | 原始数据如何进来 | #1 #2 #3 |
| **Dataset → DatasetVersion** | 版本化资产(重心);版本不可变 | #13–#20 |
| **Job** | **唯一**改动数据的东西——读版本、跑 DJ、写新版本 | #6 #7 #8 #9 #10 #14 #19 |
| **Lineage** | 由 Job 的边自然导出,不是独立功能 | #11 |

两个**不**适配该模型、单列子系统的东西:

- **安全 / RBAC + 审计**(#5,以及 #4 的"留记录"那一半)—— 横切,包住一切。
- **数据标注**(#12)—— 人在环中的工作流,有自己的状态机。

只要把 **Job → DatasetVersion → Lineage** 这个三角做对,则"清洗 / 质量 / 合成 / 处理 / 去重 / 自动打标 / 安全审核"就是**同一套机器换不同 YAML**;而可溯源(#11)是**免费**的——每个版本都记录了产出它的 job 和该 job 消费的版本。

### 四条不可动摇的设计规则

1. **版本不可变;Job 是唯一写入者。** 任何"编辑 / 清洗 / 删低质 / 合成"(#6 阈值删除、#7 采纳持久化、#8 合成、#19 增删改查)都是:**跑一个 Job → 产出一个新 DatasetVersion**。没有原地修改。这一条同时满足 #6/#8/#9/#19 反复出现的"存为新数据集或新版本",并为 #11 提供可复现性。
2. **Job 是通用的;`type` + `config_yaml` 做特化。** 今天的 `ingest_task` 收编为 `Job` 的一种 `type`。质量打分、清洗、合成、COT/QA、内容安全扫描、自动打标都是 Job,其 `config_yaml` 是一份 DJ recipe。这让"任务广场 / 重跑 / 停止 / 检索"(#10)对所有任务一视同仁,也是拖拽画布(#10)编译的目标产物。
3. **血缘不是功能,是一次查询。** 因为 `Job.inputs = [version...]` 且 `version.produced_by = job`,血缘图(#11)就是对已存边的递归遍历。无需独立血缘子系统。
4. **stats 跟着版本走。** DJ 产出 `*_stats.jsonl`(逐条得分)。把 `stats_uri` 挂在版本上,即可驱动 #6 的"查看每条数据得分详情"与阈值删除,零额外管道。

## 1. 核心领域模型

```
DataSource ──┐
             │ (ingest Job)
             ▼
Dataset 1───* DatasetVersion ◄──── produced_by ──── Job
  │  tags(#14) │  不可变快照                          │ type: ingest|clean|quality|
  │  class(#15)│  自有元数据(size/rows/...)(#13)      │       synth|process|safety|annotate
  │  lifecycle │  storage_uri (managed | S3-hosted)   │ config_yaml (DJ recipe)
  │   (#19)    │  stats_uri (逐条得分)(#6)            │ inputs: [DatasetVersion]  ← 血缘(#11)
  │  owner/ACL │  origin: managed | hosted(#18)       │ state: pending→running→success/failed
  │   (#5/#19) └─────────────────────────────────────┤ logs_uri, progress, error
  └─ valid_until(#19 生命周期)                         │ created_by, created_at(#10)
                                                       │
OperatorCatalog (源自 DJ ops/) ── 供给 ─► Job.config_yaml ◄── 拖拽编排(#10)

横切:User/Role/Permission(#5) · AuditRecord(#4 留记录/#5 审计)
旁路子系统:AnnotationProject/Task/Assignment/Result(#12)
```

### 1.1 表清单(由现状 3 表 → 约 10 表)

| 表 | 关键字段 | 需求 |
|----|---------|------|
| `data_source` | type, config(JSON), connector, status | #1 #2 #3 |
| `dataset` | name, description, sensitivity_level, business_category, owner_id, valid_until | #13 #15 #19 |
| `dataset_version` | dataset_id, version_no, storage_uri, stats_uri, rows, size, origin(managed/hosted), produced_by_job_id, **immutable** | #13 #17 #18 |
| `job` | type, name, state, config_yaml, progress, logs_uri, error, created_by | #6–#10 |
| `job_input` | job_id, dataset_version_id(多对多) | #11 |
| `operator` | key, category, param_schema(JSON), source=DJ | #7 #8 #9 #10 |
| `tag` / `tag_library` | name, library_id;dataset↔tag 多对多 | #14 |
| `audit_record` | actor, action, target, detail(JSON), created_at(**append-only**) | #4 #5 |
| `user` / `role` / `permission` | RBAC 三表 + dataset ACL | #5 #19 |
| `annotation_*` | project/task/assignment/result + 状态机 | #12 |

> 现有 `datasource` 表基本可留;`ingest_task` 收编进 `job`(`type=ingest`);`upload` 退化为 ingest job 的一种输入方式或保留为薄记录。

### 1.2 Job 状态机(M0 关键路径)

```
            ┌─────────┐  runner 取 → 起子进程        ┌──────────┐
  POST job  │ pending │ ───────────────────────────► │ running  │
 ─────────► └─────────┘                              └────┬─────┘
                                          dj-process 退出码0 │ │ 非0 / 信号
                                                            ▼ ▼
                                              ┌─────────┐  ┌────────┐
                                              │ success │  │ failed │
                                              └────┬────┘  └────────┘
                                                   │ 写出
                                                   ▼ 新建 DatasetVersion(stats_uri/storage_uri)
  停止(#10): running → 杀子进程 → cancelled
  重跑(#10): 任意终态 → 复制 config_yaml → 新建 pending job(原 job 不变,血缘可比)
```

执行落地见 [`02` §3.1 演示版架构](02-data-juicer-集成与并发调研.md):`asyncio` 子进程池 + 信号量限并发,`dj-process --config job.yaml`。

## 2. 各需求如何落到模型上

### 数据接入(#1–3)
> **落地机制是底座,连接器广度不是。** P0(见 [`00` §二](00-实现任务清单.md)):接入落地契约(任意源 → 规范化 jsonl/csv → 真实 `DatasetVersion`)+ 第一条真连接器(本地上传)——引擎的输入前提。P1/P2:其余连接器(S3/HDFS/8 库/API 推送/采集调度)是在落地机制上扩展广度,多数走"承诺支持"。
- `DataSource` 增加**可插拔 connector 接口** `fetch() -> rows`。真连接器:本地上传(已完成,P0 接成落地路径)、S3/OSS/OBS、HDFS。
- 8 种国产库走同一接口 + 驱动可得性标志 → 有驱动则真连,无驱动则脚手架 UI+API +「承诺支持」(符合需求"承诺支持即可"口径)。
- 文档解析(PDF/Office→文本,markitdown)作为 **ingest Job 的前置步骤**,#3 复用 Job 机器。
- #2 类型体系(多模态/COT/QA/偏好/时序/GIS)是 `data_type` 枚举 + JSONL 字段约定,**不新增表**。

### 内容安全(#4)
- `type=safety` 的 Job:DJ 毒性 Filter + LLM 审核 + 自定义规则/敏感词库 → 逐条判定 → 打标或删除,**并向不可变 `audit_record` 追加记录**(自研的"需要有记录"部分)。
- PII 检测(Presidio)是同一 job 里的一条可选规则。

### 数据质量(#6)
- `type=quality` Job:Filter stats 逐条打分 + Analyzer 报告;`stats_uri` 驱动逐条详情;阈值删除 = 再跑一个 job 产新版本。可视化复用 `/dj-web` analyze 视图思路。

### 数据加工(#7 清洗 / #8 合成 / #9 处理)
- 全是 Job:清洗用内置 Mapper/Deduplicator;合成/增强/蒸馏用 Mapper(原始 vs 合成靠字段标记 + 血缘区分);#9 QA 抽取(内置算子)+ COT 生成(LLM Mapper)。
- "预览→采纳→持久化"(#7):预览 = 采样跑 demo;采纳 = 正式 job 产新版本。

### 数据任务与血缘(#10/#11)
- #10 任务广场/重跑/停止/删除/检索 = 对统一 `job` 表的操作,所有 type 一视同仁。
- #10 拖拽编排 = 画布 → `config_yaml`(经 YAML 生成器,依赖 `operator` 目录)。
- #11 血缘 = 递归遍历 `job_input` 与 `produced_by_job_id` 的边。

### 数据集仓库(#13–20)—— 全是 Dataset/Version 的属性或操作
- #13 元数据 = 固化列 + 可变列;#14 标签 + `tag_library`,自动打标 = `type=process` 分类 job;#15 = sensitivity_level × business_category 两列;#16 = 元数据过滤查询 +(加分)embedding 内容检索;#17 版本 = 核心模型本身;#18 托管 = `origin=hosted` 的版本,读 S3 但写/删被门控;#19 = 删除/批删/共享+ACL/导出到本地或 S3/生命周期(valid_until + 清扫 job);#20 预置银行业数据集 = DJ 合成补齐 ≥2000 条/集。

### 数据标注(#12)
- 旁路子系统:`AnnotationProject / Task / Assignment / Result` 自有状态机(分配→标注→审核),LLM 预标注是一个 Job,交叉比例是配置。
- 标注项目完成的**产物就是一个新的 DatasetVersion** —— 重新汇入核心模型。

### 访问安全(#5)
- 横切 `User / Role / Permission` + 中间件替换 mock 登录,每个写操作端点门控,每个动作 → `audit_record`。

## 3. 后端分层(在现状上扩展)

```
app/
  models/      datasource, dataset, dataset_version, job, job_input, operator,
               tag, audit, annotation, user/role             # 现 3 表 → ~10 表
  services/
    connectors/   s3, hdfs, db_*, upload, hosted_s3           # fetch() 接口
    engine/       yaml_builder, runner(子进程池), stats_parser, lineage   # P0 地基
    catalog/      operator catalog(源自 DJ ops/,可借知识图谱)
    safety/ quality/ synth/ annotation/                       # 薄:各自只组装一个 Job
    security/     rbac, audit
  api/v1/      + datasets, jobs, operators, tags, lineage, annotation, security
```

前端 8 个占位页(security/quality/processing/data-tasks/lineage/annotation/datasets·list/datasets·presets)与这些新 API 组 1:1 对应。

## 5. 完整实现 vs 6/30 演示版

本设计是**完整目标**。模型在两版之间**不变**,只是连接器数量与前端精修的"真实度"不同:

| 维度 | 演示版(6/30) | 完整版 |
|------|-------------|--------|
| connector | 上传 + S3 真实;DB/HDFS 脚手架+承诺 | 全部真实驱动 + 重试/断点 |
| 标注 | 基础单/多人 + LLM 预标注 | 复杂审核流 / Label Studio 深度集成 |
| 内容检索(#16) | 元数据检索 | + embedding 向量检索 |
| 预置集(#20) | DJ 合成补齐 | 行业真实数据集采购/清洗 |
| 执行 | 单机 np 子进程池 | 大数据集切 Ray 集群(见 02 生产版) |

> 演示版裁剪明细见 [`00-实现任务清单.md` §六](00-实现任务清单.md)。
