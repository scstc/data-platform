# 05 · 数据安全与合规

> 安全不是流水线的某一站，而是贯穿全程的横切关注点。权威综述把它定义为数据质量的 Sanitization（净化）维度。

## Sanitization 维度覆盖什么（核验 3-0 ✓）

来源：[Data×LLM](https://arxiv.org/html/2505.18458v3) 的 IaaS 框架

| 类别 | 内容 | 对应本项目需求 |
|------|------|--------------|
| **隐私** | 排除 PII（个人可识别信息）与地理位置元数据 | 需求 4「敏感数据」 |
| **毒性** | 仇恨言论、暴力煽动（即需求里的「黄赌毒政恐」） | 需求 4「内容审核」 |
| **伦理一致性** | 极端主义、违法指引、刻板印象 | 需求 4 |
| **错误信息** | 未核实的医疗主张、政治误导 | 需求 4「数据投毒」相关 |

## PII 识别与脱敏

**Microsoft Presidio**（[github.com/microsoft/presidio](https://github.com/microsoft/presidio/)）是该领域最常被引用的开源引擎：NER + 规则 + 校验和（如银行卡 Luhn 校验）混合识别，支持打标/遮蔽/替换/加密多种脱敏动作，可扩展自定义实体——银行场景的卡号、账号、身份证号正是其典型用例。

NeMo Curator 的六阶段管线把 **PII redaction** 内置在第 5 阶段（模型质量过滤同阶段，核验 3-0 ✓）；Dataverse 也内置 PII 移除算子（核验 3-0 ✓）。换句话说：**业界标准管线里 PII 处理是必选项，不是可选项**。

## 有毒内容检测

实践路径（按成本）：

1. **词表/规则**：第一道闸，召回高误杀也高
2. **专用分类模型**：Detoxify、Perspective API 一类毒性打分模型
3. **LLM 审核**：用大模型按自定义提示词审核（即需求 4 的「利用大模型对数据内容进行审核」），可输出理由，支持打标或删除并留记录

> ⚠️ 诚实声明：Detoxify / Perspective API / Llama Guard 的能力与准确率对比**本次调研未独立核验**（属综述级提及），选型前需补充评测。

## 数据投毒（Data Poisoning）防御 `待补研`

概念：攻击者向训练数据注入恶意样本（如后门触发词），使模型在特定输入下产生攻击者期望的行为。与「数据里碰巧有脏东西」的区别在于**对抗性**。

本次调研确认了该威胁在业界被严肃对待（来源：[Lakera 博客](https://www.lakera.ai/blog/training-data-poisoning)，blog 级来源），但**具体防御方法（spectral signatures、影响函数、后门检测）缺一手核验**，已列入开放问题。当前可执行的工程缓解：

- 数据来源白名单 + 接入侧记录完整血缘（能追溯每条数据从哪来）
- 内容审核管线对全部入库数据无差别过滤
- 训练前抽样人审 + 训练后行为评测

## 数据访问安全

原始需求 5「严格数据权限管理，无越权访问风险」属于平台工程而非数据工程算法——RBAC/行级权限/审计日志，由数据集仓库层实现（见 [06](06-数据标注与数据集管理.md)），不依赖模型。

## 用什么模型（速查）

| 任务 | 推荐 |
|------|------|
| PII 识别 | Presidio（NER+规则混合），中文实体需自定义扩展 |
| 毒性检测 | 专用分类器粗筛 + LLM 复核（自定义提示词、可留审核记录） |
| 投毒防御 | 暂无成熟单点工具，靠血缘+审核+评测组合缓解（待补研） |
