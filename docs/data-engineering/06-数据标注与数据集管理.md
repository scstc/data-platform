# 06 · 数据标注与数据集管理

> ⚠️ 诚实声明：本篇两个主题在本次调研中**一手证据较薄**（综述层面提及，缺工具级独立核验），标注 `待补研` 的部分请以官方文档与后续补研为准。本篇仍值得读——它框定了概念和选型方向。

## 第一部分：数据标注

### 标注什么

| 训练场景 | 标注形态 |
|---------|---------|
| SFT | 给指令写高质量响应 / 修正模型草稿 |
| RLHF/DPO | 对同一 prompt 的多个回复**排序**（如 UltraFeedback 6.4 万条，核验 3-0 ✓） |
| 分类/审核 | 给样本打标签（毒性、质量分、领域） |

### 业界主流模式：智能预标注 + 人工审核

纯人工标注太贵，标准做法是机器先标、人只审：

1. **智能标注**：LLM/专用模型生成初始标签（即 FineWeb-Edu 的 LLM 打分思路，那是被核验过的真实案例）
2. **人工审核**：单人审或多人交叉审（配置交叉比例对应需求 12 的「交叉比例」概念——同一样本分给多人标，用一致性衡量质量）
3. **进度与报告**：标注平台自带

### 工具 `待补研`

**Label Studio** 是开源标注平台的事实标准（多模态、多人协作、ML 后端预标注）。值得注意：data-juicer 仓库根目录就有 `label_studio_localhost_connection.json`，上游已有集成迹象，落地时优先验证这条通路。本次调研未对 Label Studio 能力做独立核验。

## 第二部分：数据集管理（数据集仓库）

对应原始需求 13-20 的全部内容。这一层不是算法问题，是**平台工程**问题。

### 概念速览

| 概念 | 含义 | 对应需求 |
|------|------|---------|
| **元数据** | 固化信息（来源/类型/大小/创建人/时间/版本）+ 可变信息（名称/描述/有效期） | 需求 13 |
| **版本** | 同一数据集多版本并存，各带元数据；处理操作产出新版本而非覆盖 | 需求 17 |
| **血缘（lineage）** | 记录「这个数据集由哪个任务、从哪些上游数据、用哪些算子加工而来」，保证可溯源可复现 | 需求 11 |
| **分级分类** | 按敏感级别（公开/内部/机密）× 业务类型双维度管理 | 需求 15 |

### 为什么血缘特别重要

它是多个需求的交汇点：可复现性（重跑任务得到同样数据集）、投毒溯源（发现坏数据能找到入口）、合规审计（证明训练数据来源合法）。工程实现的关键是：**每个处理任务记录 输入数据集版本 + 算子配置（YAML）+ 输出版本** 的三元组——data-juicer 的配置驱动模式（一个 YAML 完整描述一次处理）天然适合做血缘的「算子配置」环节。

### 工具方向 `待补研`

数据版本控制常被提及的有 DVC、lakeFS（Git 式数据版本管理思路）；本次调研未覆盖其在 LLM 数据平台的实践细节，已列入开放问题。HuggingFace Datasets 的 dataset card + revision 机制可作为元数据/版本设计的参考原型。

## 落地建议（基于已核验事实的推断）

数据集仓库层 data-juicer **不提供**（它是处理引擎不是资产管理平台），需自建：元数据库（PostgreSQL 足矣）+ 对象存储（S3/OSS，对应需求 1/18 的托管）+ 任务记录表构成血缘。这是本项目相对 data-juicer 的主要自研增量，详见 [08-需求映射](08-需求映射.md)。
