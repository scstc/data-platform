# 01 · LLM 数据工程生命周期总览

> 一句话：大模型的能力上限很大程度由数据决定（Data-Centric AI），数据工程就是把原始数据变成「能喂给模型的高质量数据集」的全部工程化流程。

## 为什么需要数据工程

直接从网上爬来的数据充满重复、乱码、广告、有毒内容和隐私信息。直接训练的后果：浪费算力、模型学坏（毒性/偏见）、法律风险（PII/版权）。所以业界形成了一条标准化的处理流水线。

## 生命周期七阶段

| 阶段 | 做什么 | 典型方法/模型 | 详见 |
|------|--------|--------------|------|
| 1. 数据接入/采集 | 从对象存储、数据库、网页、API 拉取原始数据，统一为标准格式（如 JSONL） | CommonCrawl 抽取、trafilatura 正文抽取 | — |
| 2. 数据清洗 | 去乱码、规范空格、去 HTML 标签、语言识别分流 | 规则算子 + fastText 语言识别 | [02](02-数据清洗与去重.md) |
| 3. 数据去重 | 删掉重复/近似重复的样本 | 精确 hash / MinHash+LSH / 语义 embedding | [02](02-数据清洗与去重.md) |
| 4. 质量评估 | 给每条数据打分，过滤低质量 | 困惑度、fastText/BERT 分类器、LLM-as-a-Judge | [03](03-数据质量评估.md) |
| 5. 合成与增强 | 数据不够时用 LLM 造数据、增强已有数据、蒸馏 | Self-Instruct / Evol-Instruct / Magpie | [04](04-数据合成与增强.md) |
| 6. 数据标注 | 人工/智能标注监督信号 | Label Studio、智能预标注+人工审核 | [06](06-数据标注与数据集管理.md) |
| 7. 数据集管理 | 元数据、版本、血缘、分级分类 | 数据集仓库、lineage 追踪 | [06](06-数据标注与数据集管理.md) |

**数据安全与合规**（去毒、PII 脱敏、防投毒）不是独立阶段，而是贯穿全程的横切关注点 → [05](05-数据安全与合规.md)。

## 怎么衡量一个数据集「好」：IaaS 四维框架

权威综述 [Data×LLM](https://arxiv.org/html/2505.18458v3) 提出 **IaaS** 四维（核验 3-0 ✓）：

| 维度 | 含义 |
|------|------|
| **I**nclusiveness 包容性 | 覆盖足够广的领域、语言、风格 |
| **A**bundance 丰富性 | 数量足够、分布合理 |
| **A**rticulation 清晰表达 | 文本通顺、结构清楚 |
| **S**anitization 净化 | 严格过滤 PII/隐私、有毒内容、伦理不一致、错误信息 |

## 不同训练阶段，数据需求完全不同

来源：[Data×LLM](https://arxiv.org/html/2505.18458v3)（核验 3-0 ✓）

| 场景 | 数据形态 | 规模感 | 关键要求 |
|------|---------|--------|---------|
| **预训练** | 网页/代码/书籍等多样语料 | **TB 级** | 多样性 + 大规模去重去噪 |
| **SFT 微调** | 指令-响应对 | 数千~数百万条（如法律领域 DISC-Law-SFT 29.5 万条） | 指令多样、响应高质量 |
| **RLHF/偏好（DPO）** | 同一 prompt 的多个回复 + 人工排序 | 比 SFT 更小（如 UltraFeedback 6.4 万条） | 排序一致性、标注质量 |
| **RAG** | 参考语料库 | 大规模 | **严格审核** + 支持实时动态更新 |

> 💡 对照本项目：原始需求里的「COT 思维数据 / 问答对数据 / 偏好数据」就分别对应 SFT 和偏好对齐场景的数据形态，见 [08-需求映射](08-需求映射.md)。

## 和 data-juicer 的关系

data-juicer 的算子体系正是按这条生命周期组织的：**Formatter**（接入/格式统一）→ **Mapper**（清洗/编辑）→ **Filter**（质量过滤）→ **Deduplicator**（去重）→ **Selector**（按需采样）。详见 [07-工具生态对比](07-工具生态对比.md)。
